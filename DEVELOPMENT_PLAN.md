# 逆命仙途前后端开发计划

## 一、项目目标

制作一个以文字事件、属性成长和随机选择为核心的修仙肉鸽游戏。

单局目标时长为 20～40 分钟。玩家通过事件选择、功法组合和资源管理完成一轮修行；死亡或飞升后结算，并使用因果点解锁新的开局内容。

技术栈固定为：

- 后端：Java 21+、Spring Boot、Spring Data JPA
- 数据库：MySQL 8
- 前端：React、TypeScript、Vite
- 测试：Spring Boot Test、H2、前端 TypeScript 构建
- 部署：Nginx、Docker Compose

## 二、当前完成情况

当前原型已经具备：

- React 开始游戏页面和事件选择页面
- Spring Boot 游戏接口
- MySQL `game_run` 存档表
- 创建旅程、查询旅程、提交事件选择
- 服务端计算属性变化
- H2 接口自动化测试
- MySQL 初始化脚本和 Docker Compose 配置

当前接口：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| POST | `/api/game/runs` | 创建一局游戏 |
| GET | `/api/game/runs/{id}` | 查询游戏存档 |
| POST | `/api/game/runs/{id}/choices` | 提交事件选择 |

## 三、版本路线

### V0.1：当前原型

目标：证明核心玩法和前后端链路可运行。

- 角色创建
- 事件三选一
- 气血、灵力、寿元、因果变化
- MySQL 保存游戏状态
- React 基础游戏界面

状态：已完成。

### V0.2：可完整通关的单局游戏

目标：玩家可以完整体验一局“开始—选择—结局”。

状态：第一批任务已完成，当前单局默认路线约 12 次决策，后续继续扩充内容和数值平衡。

后端：

- 增加 10～15 个事件
- 增加事件链和分支条件
- 增加死亡、飞升、失败结局
- 增加服务端随机种子
- 增加回合结算日志
- 增加游戏状态校验

前端：

- 显示完整属性面板
- 增加事件历史记录
- 增加结局页面
- 增加重新开始按钮
- 增加加载中、错误和断线提示

验收标准：

- 新玩家能完成至少 8 个事件
- 至少出现 1 个死亡结局和 1 个飞升结局
- 刷新页面后可以根据游戏 ID 恢复进度

### V0.3：账号与角色系统

目标：游戏存档从临时 ID 变成用户账号下的角色存档。

后端：

- 用户注册、登录、退出
- Spring Security + JWT
- 密码哈希存储，禁止明文密码
- 用户与角色绑定
- 查询最近游戏记录
- 限制只能访问自己的存档

前端：

- 登录页
- 注册页
- 角色选择页
- 登录态保存和过期处理
- 未登录访问游戏页时跳转登录

新增接口：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| POST | `/api/auth/register` | 注册账号 |
| POST | `/api/auth/login` | 登录并返回 Token |
| GET | `/api/players/me` | 查询当前角色 |
| GET | `/api/game/runs` | 查询当前用户的游戏记录 |

### V0.4：功法、法宝和肉鸽构筑

目标：让不同选择形成可重复游玩的流派。

首批流派建议：

- 剑修：高爆发、高风险
- 丹修：恢复和资源管理
- 体修：高气血、低灵力
- 鬼修：牺牲寿元换取强力效果

后端：

- 功法配置
- 法宝配置
- 天赋配置
- 被动效果和触发条件
- 事件奖励
- 构筑结算
- 因果点和永久解锁

前端：

- 功法面板
- 法宝和背包面板
- 当前构筑预览
- 奖励选择弹窗
- 死亡结算与永久解锁页面

验收标准：

- 至少 4 个流派有明显差异
- 同一事件在不同构筑下能产生不同结果
- 单局至少存在 2 种有效通关思路

### V0.5：内容配置和运营基础

目标：让新增内容不必频繁修改 Java 核心代码。

- 将事件、功法、法宝抽离为 MySQL 配置
- 增加配置版本号和启用状态
- 增加后台导入 JSON 配置能力
- 增加配置校验
- 增加操作日志
- 增加简单排行榜

Redis 在这个阶段再加入，用于排行榜和热点配置缓存；第一版不提前引入微服务或复杂消息队列。

### V1.0：可发布版本

- 完整账号和存档系统
- 至少 50 个事件
- 至少 10 个功法或天赋
- 至少 20 件法宝或消耗品
- 至少 6 种结局
- 移动端适配
- API 鉴权和参数校验
- 后端单元测试、接口测试和 MySQL 集成测试
- Docker Compose 一键启动
- 生产环境日志和异常监控

## 四、数据库规划

当前已有：

- `game_run`：一局游戏的实时状态

后续表：

| 表 | 作用 |
| --- | --- |
| `app_user` | 账号、密码哈希、登录信息 |
| `player` | 用户角色、境界、永久成长 |
| `game_run` | 当前一局游戏状态 |
| `run_event` | 本局每次事件和选择记录 |
| `event_config` | 事件内容、条件和选项配置 |
| `skill_config` | 功法、天赋和被动效果 |
| `item_config` | 法宝、丹药和消耗品 |
| `player_skill` | 角色拥有的功法 |
| `player_item` | 角色拥有的物品 |
| `run_settlement` | 死亡或飞升结算记录 |
| `unlock_record` | 因果点解锁内容 |

数据库迁移建议：

- V0.2 前引入 Flyway
- 正式环境关闭 `ddl-auto: update`
- 所有表结构变更使用版本化 SQL
- 关键数据使用事务保存

## 五、前后端协作规则

### 后端规则

- 随机数、属性计算、奖励结算全部由服务端负责
- 客户端只能提交选择编号，不能提交最终属性
- 每局保存 `seed`，支持复盘和问题定位
- `game_run.version` 用于乐观锁，防止重复提交覆盖状态
- 事件提交需要幂等控制，避免用户重复点击造成重复结算
- 所有接口统一返回错误结构

### 前端规则

- 使用 TypeScript 类型描述 API 数据
- API 请求集中放在 `src/api.ts`
- 页面只负责展示和提交用户选择
- 统一处理加载、空数据、网络错误和登录过期
- 事件选择按钮提交后立即禁用，等待服务端返回
- 移动端优先设计，保证手机宽度下可操作

## 六、建议开发顺序

第一阶段先完成 V0.2：

1. 把当前事件配置拆成独立结构
2. 增加事件链、结局和完整日志
3. 完成存档恢复
4. 完善 React 游戏页面
5. 增加后端接口测试

第二阶段完成 V0.3：

1. Flyway 数据库迁移
2. 用户注册和登录
3. JWT 鉴权
4. 用户角色和存档绑定
5. React 登录态管理

第三阶段完成 V0.4：

1. 功法、法宝和天赋模型
2. 构筑效果计算
3. 奖励和结算页面
4. 因果点永久成长

## 七、第一批任务拆分

建议下一次开发直接处理以下 5 个任务：

- [x] 增加 `run_event` 表和事件历史记录
- [x] 将事件定义从 `GameService` 拆到独立配置类
- [x] 增加完整结局判断
- [x] 增加 GET 存档恢复的前端入口
- [x] 增加重复提交和非法选项测试

完成这 5 项后，项目就从“技术演示”进入“可以实际玩一局”的阶段。
