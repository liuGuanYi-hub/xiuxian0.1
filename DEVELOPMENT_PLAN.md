# 逆命仙途前后端开发计划

## 一、项目目标

制作一个以文字事件、属性成长和随机选择为核心的修仙肉鸽游戏。

单局目标时长为 20～40 分钟。玩家通过事件选择、功法组合和资源管理完成一轮修行；死亡或飞升后结算，并使用因果点解锁新的开局内容。

技术栈固定为：

- 后端：Java 21+、Spring Boot、Spring Data JPA
- 数据库：MySQL 8
- 前端：React、TypeScript、Vite
- 测试：Spring Boot Test、H2、前端 TypeScript 构建
- 部署：Nginx、Docker Compose

## 二、当前完成情况

当前原型已经具备：

- React 开始游戏页面和事件选择页面
- 每局 seed 生成 10 层路线图，节点之间保存相邻连线和解锁状态
- 多种节点类型：普通战斗、精英战斗、随机事件、休息闭关、坊市商店、秘境宝藏、渡劫 Boss
- 事件内容、稀有事件和可重复事件的权重池
- Spring Boot 游戏接口
- MySQL `game_run` 存档表
- MySQL `run_map_node` 路线节点表
- MySQL `run_build_item` 本局构筑表和 `run_reward_offer` 待选奖励表
- 创建旅程、查询旅程、提交事件选择
- 进入节点、完成节点并解锁下一跳路线
- 初始功法/符箓构筑，战斗、精英、秘境节点奖励三选一
- 奖励卡的即时属性效果和后续战斗加成
- 服务端计算属性变化
- 多种特殊结局：天门飞升、红尘道君、逆命魔尊、逍遥散仙、断因绝果、道途断绝
- H2 接口自动化测试
- MySQL 初始化脚本和 Docker Compose 配置

当前接口：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| POST | `/api/game/runs` | 创建一局游戏 |
| GET | `/api/game/runs/{id}` | 查询游戏存档 |
| POST | `/api/game/runs/{id}/nodes/{nodeId}/enter` | 进入当前可达节点 |
| POST | `/api/game/runs/{id}/choices` | 提交事件选择 |
| POST | `/api/game/runs/{id}/rewards/{rewardId}/claim` | 领取构筑奖励并解锁下一跳 |

## 三、版本路线

### V0.1：当前原型

目标：证明核心玩法和前后端链路可运行。

- 角色创建
- 事件三选一
- 气血、灵力、寿元、因果变化
- MySQL 保存游戏状态
- React 基础游戏界面

状态：已完成。

### V0.2：可完整通关的单局游戏

目标：玩家可以完整体验一局“开始—选择—结局”。

状态：路线图重构已完成，当前单局最多经过 10 层节点并抵达渡劫 Boss；后续继续扩充构筑内容和数值平衡。

后端：

- 增加 10～15 个事件
- 增加事件链和分支条件
- 使用本局 seed 生成多分支路线图，保证不同局路线不同、同一局可以复盘
- 按节点类型和权重抽取节点内容，支持稀有事件和可重复节点
- 增加随机机缘和随机反噬，并写入事件历史
- 增加死亡、飞升、失败结局
- 增加服务端随机种子
- 增加回合结算日志
- 增加游戏状态校验

前端：

- 显示完整属性面板
- 增加事件历史记录
- 增加结局页面
- 增加重新开始按钮
- 增加加载中、错误和断线提示

验收标准：

- 新玩家能完成至少 8 个事件
- 至少出现 1 个死亡结局和 1 个飞升结局
- 刷新页面后可以根据游戏 ID 恢复进度

### V0.3：账号与角色系统

目标：游戏存档从临时 ID 变成用户账号下的角色存档。

后端：

- Spring Security + JWT

- 用户与角色绑定
- 查询最近游戏记录
- 限制只能访问自己的存档

前端：

- 登录页
- 注册页
- 角色选择页
- 登录态保存和过期处理
- 未登录访问游戏页时跳转登录

新增接口：

| 方法 | 路径 | 作用 |

- [x] 战斗快照：保存敌方气血、护盾、蓄力、中毒、玩家护盾/中毒、回合和战斗日志

- [x] 战斗流程：战斗胜利进入奖励三选一，Boss 胜利进入结局，战斗失败进入死亡结算
- [x] 构筑接入：伤害、护盾、调息灵力和战技中毒加成由 ACTIVE 卡牌和流派协同提供

## 五、前后端协作规则

### 后端规则

- 随机数、属性计算、奖励结算全部由服务端负责
- 客户端只能提交选择编号，不能提交最终属性
- 每局保存 `seed`，支持复盘和问题定位
- `game_run.version` 用于乐观锁，防止重复提交覆盖状态
- 事件提交需要幂等控制，避免用户重复点击造成重复结算
- 所有接口统一返回错误结构

### 前端规则

- 使用 TypeScript 类型描述 API 数据
- API 请求集中放在 `src/api.ts`
- 页面只负责展示和提交用户选择
- 统一处理加载、空数据、网络错误和登录过期
- 事件选择按钮提交后立即禁用，等待服务端返回
- 移动端优先设计，保证手机宽度下可操作
