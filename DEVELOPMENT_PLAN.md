# 逆命仙途前后端开发计划

## 一、项目目标

制作一个以文字事件、属性成长和随机选择为核心的修仙肉鸽游戏。

单局目标时长为 20～40 分钟。玩家通过事件选择、功法组合和资源管理完成一轮修行；死亡或飞升后结算，并使用因果点解锁新的开局内容。

技术栈固定为：

- 后端：Java 21+、Spring Boot、Spring Data JPA
- 数据库：MySQL 8
- 前端：React、TypeScript、Vite
- 测试：Spring Boot Test、H2、前端 TypeScript 构建
- 部署：Nginx、Docker Compose

## 二、当前完成情况

当前原型已经具备：

- React 开始游戏页面和事件选择页面
- 每局 seed 生成 10 层路线图，节点之间保存相邻连线和解锁状态
- 多种节点类型：普通战斗、精英战斗、随机事件、休息闭关、坊市商店、秘境宝藏、渡劫 Boss
- 事件内容、稀有事件和可重复事件的权重池
- Spring Boot 游戏接口
- MySQL `game_run` 存档表
- MySQL `run_map_node` 路线节点表
- MySQL `run_build_item` 本局构筑表和 `run_reward_offer` 待选奖励表
- 创建旅程、查询旅程、提交事件选择
- 进入节点、完成节点并解锁下一跳路线
- 初始功法/符箓构筑，战斗、精英、秘境节点奖励三选一
- 奖励卡的即时属性效果和后续战斗加成
- 服务端计算属性变化
- 多种特殊结局：天门飞升、红尘道君、逆命魔尊、逍遥散仙、断因绝果、道途断绝
- H2 接口自动化测试
- MySQL 初始化脚本和 Docker Compose 配置

当前接口：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| POST | `/api/game/runs` | 创建一局游戏 |
| GET | `/api/game/runs/{id}` | 查询游戏存档 |
| POST | `/api/game/runs/{id}/nodes/{nodeId}/enter` | 进入当前可达节点 |
| POST | `/api/game/runs/{id}/choices` | 提交事件选择 |
| POST | `/api/game/runs/{id}/rewards/{rewardId}/claim` | 领取构筑奖励并解锁下一跳 |

## 三、版本路线

### V0.1：当前原型

目标：证明核心玩法和前后端链路可运行。

- 角色创建
- 事件三选一
- 气血、灵力、寿元、因果变化
- MySQL 保存游戏状态
- React 基础游戏界面

状态：已完成。

### V0.2：可完整通关的单局游戏

目标：玩家可以完整体验一局“开始—选择—结局”。

状态：路线图重构已完成，当前单局最多经过 10 层节点并抵达渡劫 Boss；后续继续扩充构筑内容和数值平衡。

后端：

- 增加 10～15 个事件
- 增加事件链和分支条件
- 使用本局 seed 生成多分支路线图，保证不同局路线不同、同一局可以复盘
- 按节点类型和权重抽取节点内容，支持稀有事件和可重复节点
- 增加随机机缘和随机反噬，并写入事件历史
- 增加死亡、飞升、失败结局
- 增加服务端随机种子
- 增加回合结算日志
- 增加游戏状态校验

前端：

- 显示完整属性面板
- 增加事件历史记录
- 增加结局页面
- 增加重新开始按钮
- 增加加载中、错误和断线提示

验收标准：

- 新玩家能完成至少 8 个事件
- 至少出现 1 个死亡结局和 1 个飞升结局
- 刷新页面后可以根据游戏 ID 恢复进度

### V0.3：账号与角色系统

目标：游戏存档从临时 ID 变成用户账号下的角色存档。

后端：

- 用户注册、登录、退出
- Spring Security + JWT
- 密码哈希存储，禁止明文密码
- 用户与角色绑定
- 查询最近游戏记录
- 限制只能访问自己的存档

前端：

- 登录页
- 注册页
- 角色选择页
- 登录态保存和过期处理
- 未登录访问游戏页时跳转登录

新增接口：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| POST | `/api/auth/register` | 注册账号 |
| POST | `/api/auth/login` | 登录并返回 Token |
| GET | `/api/players/me` | 查询当前角色 |
| GET | `/api/game/runs` | 查询当前用户的游戏记录 |

### V0.4：功法、法宝和肉鸽构筑

目标：让不同选择形成可重复游玩的流派。

状态：第一批构筑闭环已完成：每局拥有 starter 卡组，战斗/精英/秘境节点结算后提供三选一奖励，卡牌效果会影响后续战斗。当前已补上灵石资源与休息/闭关节点卡牌升级，后续继续增加卡牌移除、坊市和流派协同。

下一批构筑任务：

- [x] 卡牌升级：在休息/闭关节点消耗灵石强化卡牌效果，并支持跳过升级
- [ ] 卡牌移除：在坊市或特殊事件中移除弱卡，形成更稳定的构筑
- [ ] 坊市购买与刷新：增加灵石、价格、刷新次数和稀有度定价
- [ ] 流派协同：剑修、丹修、体修、鬼修等卡牌组合产生额外效果
- [ ] 奖励池条件：根据路线、精英节点、稀有事件和当前构筑调整奖励权重
- [ ] 构筑统计：展示功法/法宝/符箓数量、流派标签和主要战斗加成
- [ ] 卡牌数据配置化：将卡牌从 Java 静态目录迁移到 `skill_config`、`item_config` 和符箓配置表

下一批验收标准：

- 玩家可以在一局内升级或移除至少一张卡牌
- 坊市可以购买卡牌并刷新商品，刷新结果受本局 seed 影响
- 至少 4 个流派拥有可识别的两卡或三卡协同效果
- 卡牌升级、移除、购买和刷新在刷新页面后不会丢失
- 同一套卡组在普通战斗、精英战斗和 Boss 前准备阶段有明显策略差异

首批流派建议：

- 剑修：高爆发、高风险
- 丹修：恢复和资源管理
- 体修：高气血、低灵力
- 鬼修：牺牲寿元换取强力效果

- [x] 卡牌即时效果和战斗被动加成
- [x] 灵石资源、卡牌强化等级和休息节点升级接口

- 天赋配置

- 事件奖励

前端：

## 五、前后端协作规则

### 后端规则

- 随机数、属性计算、奖励结算全部由服务端负责
- 客户端只能提交选择编号，不能提交最终属性
- 每局保存 `seed`，支持复盘和问题定位
- `game_run.version` 用于乐观锁，防止重复提交覆盖状态
- 事件提交需要幂等控制，避免用户重复点击造成重复结算
- 所有接口统一返回错误结构

### 前端规则

- 使用 TypeScript 类型描述 API 数据
- API 请求集中放在 `src/api.ts`
- 页面只负责展示和提交用户选择
- 统一处理加载、空数据、网络错误和登录过期
- 事件选择按钮提交后立即禁用，等待服务端返回
- 移动端优先设计，保证手机宽度下可操作
